Product Roadmap

Strata Self-Management App

A shared issue tracker for strata committees — recording open problems, tracking who's been contacted, logging activity, attaching documents, and making the state of the building visible to everyone who needs to see it.

Weeks 1–3
Phase 0 — Foundation
Scaffolding, data model, deployment pipeline. Everything downstream depends on getting this right and keeping it simple.
⚙
Stack & project setup
Pick one framework you're fast in. Server-rendered, PostgreSQL, ORM. No SPA complexity.
›
  • Rails, Django, or SvelteKit — whichever you ship fastest in
  • PostgreSQL with a conventional ORM
  • Deploy to a single managed platform (Fly.io, Railway, Render)
  • CI pipeline: push to main → deploy. Nothing fancier.
🗄
Multi-tenant data model
Buildings, users, roles, issues, activity log, issue links — all tenant-scoped from day one.
›
  • buildings — the tenant. Everything scoped to this.
  • users — email, name, role per building (chair, committee, owner)
  • issues — title, description, status, priority, category, building_id
  • activity_entries — text, timestamp, author, issue_id, optional file attachment, optional contact reference
  • issue_links — join table: issue_a, issue_b, relationship label
  • contacts — external providers/people, linked to buildings
🔐
Auth
Use a managed service — Clerk, Supabase Auth, or Auth.js. Don't build login flows from scratch.
›
  • Magic links or email/password — whatever the auth service defaults to
  • Invitation flow: chair invites committee members by email
  • Row-level tenant scoping: every query filtered by building_id
Weeks 3–8
Phase 1 — Issue Tracker
The core product. Build it, use it on your own building immediately. Documents and contacts are baked in — not separate features.
📋
Issue list & detail view
Filterable list of issues. Click through to a detail view with the full activity feed.
›
  • Status: open, waiting on us, waiting on external, resolved, closed
  • Priority: urgent, high, normal, low
  • Category: maintenance, compliance, financial, governance, other
  • Filter and sort by any combination of the above
  • Issue linking: relate issues to each other with optional label (blocks, related to, part of)
💬
Activity log
Chronological feed on each issue. Notes, status changes, contact records, file attachments — all in one stream.
›
  • Free-text note entry with timestamp and author
  • Status changes logged automatically
  • Optional contact tag on each entry ("called Dave at AED, no answer")
  • Optional file attachment — quotes, invoices, reports, photos
  • File storage: S3-compatible object storage (Cloudflare R2 or similar)
💰
Inline financial data
When attaching a quote or invoice, capture amount, supplier, status. No separate accounting module — just metadata on documents.
›
  • Document types: quote, invoice, report, certificate, correspondence, photo
  • Financial fields on quote/invoice types: amount, supplier, date, status (received / approved / paid)
  • Auto-generated summary per issue: total quoted, approved, paid, outstanding
  • No double-entry accounting — this is a transparency layer, not a ledger
👤
Contacts directory (lightweight)
External providers and their details, created inline when you tag them on an activity entry. Not a standalone CRM.
›
  • Name, company, role, phone, email
  • Created on-the-fly when tagging an activity entry
  • Contact detail page shows every issue and activity entry they're referenced in
  • Over time, builds a full history of interactions with each provider
◆
Milestone — You are using this daily on your own building. Every open issue is tracked. Your fire order has a full activity log with attached documents.
Weeks 8–10
Phase 2 — First External Users
Find 2–3 other committee chairs. Set them up manually. Charge them immediately, even if it's $10/month. Validate that this works for someone who isn't you.
🏢
Manual onboarding
You create their building, add their login, walk them through it on a call. Unscalable and intentional.
›
  • Source users from strata owner communities — Flat Chat, OC forums, Facebook groups
  • Offer to set up their building and import their current open issues for them
  • Watch them use it. Note every point of confusion. Fix the worst ones immediately.
💳
Billing — Stripe integrationRevenue
Per-building monthly subscription. One afternoon of Stripe setup. Charge from day one.
›
  • Simple tier: ~$30–50/month per building (adjust based on lot count later)
  • Free trial if needed to reduce friction, but always collect payment method
  • Stripe Checkout or Stripe Billing — don't build a custom billing UI
👥
Multi-user access
Chair invites committee members. They can view issues, add comments, upload files.
›
  • Invitation by email — generates a magic link
  • Role-based access: chair (full control), committee (comment, upload, change status)
  • Email notifications when an issue is updated or a comment is added
◆
Milestone — At least one building other than yours is using the app and paying for it. You have confirmed willingness to pay.
Weeks 10–16
Phase 3 — Iterate on Feedback
Build what your early users actually need, not what you imagine they need. This phase is deliberately unspecified — the backlog comes from real usage.
🔄
User-driven feature work
Weekly conversations with your paying users. Fix what they complain about. Build what they ask for.
›
  • Likely candidates: better notification controls, dashboard/summary view, print-friendly reports for committee meetings
  • Resist building features nobody has asked for
  • Polish the rough edges that early users tripped on
📝
Blog launchMarketing
Start publishing fortnightly. Product journey and strata domain expertise. You now have something to show.
›
  • First post: "Here's the tool I built to track our building's fire order"
  • Hosted on Ghost, Substack, or a static site — zero custom code
  • Cross-post to Flat Chat, strata Facebook groups, relevant Reddit communities
  • 500–800 words per post. Fortnightly, not weekly.
  • Write about the problem space and the product — not your specific building
Weeks 16–20
Phase 4 — Self-Service Onboarding
Remove yourself from the signup flow. A new building can sign up, create their scheme, invite their committee, and start using the app without your involvement.
🚀
Signup & building creation flow
Landing page → sign up → create building → invite committee → start tracking issues. End to end, no manual steps.
›
  • Simple landing page with clear value proposition and pricing
  • Signup creates user + building in one flow
  • Guided first-run: create your first issue, invite a committee member
  • Stripe Checkout integrated into signup
🛡
Production hardening
Backups, monitoring, error tracking, basic docs. The minimum to responsibly take money from strangers.
›
  • Automated database backups (daily)
  • Error tracking (Sentry or similar)
  • Uptime monitoring
  • Terms of service and privacy policy
  • Help docs or FAQ covering the basics
◆
Milestone — The product is commercially live. Buildings can sign up and pay without your involvement. Growth is now a distribution problem, not a development problem.
Guiding Principles
Dogfood immediately
Use the app on your own building from the end of week 3. Every feature must solve a problem you actually have this week.
Charge early
The first external building pays from day one. Even $10/month. Validates willingness to pay and changes the relationship from favour to service.
Ship ugly and functional
The fire order page is the vision for year two. The MVP is a shared issue list with an activity log. That's already more than most committees have.
Resist premature generalisation
Build for one building. Then extract patterns once the domain model is stable. Don't build meeting agendas, owner portals, or AI features yet.
Deliberately Deferred (6+ months)
Resident maintenance portal
The "report a leaky tap" flow for tenants and non-committee owners.
Meeting features
Agenda builder, resolution tracker, minutes generation. Valuable but a second product inside the product.
In-app messaging
Replacing email entirely as the communication layer. The activity log is the first step toward this.
Knowledge base
Building manual, procedures, warranties, "where things are and how they're done."
